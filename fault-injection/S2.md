# S2 — DNS Outage Burst

**Scheme label:** `DNS_TIMEOUT_BURST`  
**Tool:** `iptables` on Ubuntu hotspot laptop  
**Mechanism:** Periodically DROP all DNS traffic (port 53 UDP + TCP) sent by the Uno Q to the laptop's DNS resolver, simulating a recursive DNS server outage in repeating bursts.

---

## Overview

```
Internet ←→ Ubuntu Laptop (hotspot)
                  │  wlan NIC in AP mode
                  │  IP: 10.64.88.124 (hotspot gateway + DNS resolver)
                  ↓
             Arduino Uno Q (wlan0)
                  IP: 10.64.88.173 / .174 (DHCP)
                  DNS: 10.64.88.124 (→ laptop)
```

The Uno Q resolves DNS by sending queries to the **laptop's hotspot IP**
(confirmed from `dns_resolvers: ["10.64.88.124"]` in the telemetry data).
Dropping those queries at the laptop's `INPUT` chain causes the Uno Q's
DNS resolver to time out — exactly matching an S2 DNS outage burst.

| Parameter | Value |
|---|---|
| Burst duration (BLOCK) | 15 s |
| Gap (RESTORE) | 45 s |
| Total bursts | 10 |
| Total experiment | ~10 min |
| Expected data label | `DNS_TIMEOUT_BURST` |

## Part 1 — Ubuntu Hotspot Setup

> If the hotspot is already running and the Uno Q is connected, skip to Part 2.

### 1.1 Prerequisites

```bash
sudo apt update
sudo apt install -y iptables nmap net-tools
```

### 1.2 Create the Hotspot via nmcli (command line, persistent)

Replace `wlp3s0` with your Wi-Fi interface name (`ip link` to find it).

```bash
# Show all network interfaces
ip link show

# Create a Wi-Fi hotspot (2.4 GHz band for Uno Q compatibility)
sudo nmcli device wifi hotspot \
    ifname wlp3s0 \
    ssid "micro-uxi-lab" \
    password "microuxi123" \
    band bg

# Verify it is running
nmcli connection show --active
```

> **Note:** `band bg` forces 2.4 GHz. The Uno Q only supports 2.4 GHz.

### 1.3 Alternative: GUI setup (GNOME)

1. Open **Settings → Wi-Fi**
2. Click the menu (⋮) → **Turn On Wi-Fi Hotspot**
3. Note the SSID and password shown
4. Under **Hotspot Settings**, make sure "Share to other computers" is enabled

### 1.4 Verify IP forwarding is active

The hotspot requires IP forwarding (NetworkManager enables it automatically,
but verify):

```bash
# Check
cat /proc/sys/net/ipv4/ip_forward
# Must output: 1

# If it shows 0, enable it:
sudo sysctl -w net.ipv4.ip_forward=1
```

### 1.5 Check the hotspot interface has the correct IP

```bash
ip addr show
# Look for the hotspot interface (wlp3s0 / ap0 / wlan0)
# It should show 10.64.88.124/x (or similar 10.x range)
```

### 1.6 Confirm the hotspot NAT rule exists

```bash
sudo iptables -t nat -L POSTROUTING -n --line-numbers
# You should see a MASQUERADE rule for the hotspot subnet (10.64.88.0/x)
# NetworkManager adds this automatically
```

## Part 5 — Sensor Configuration for S2

On the **Uno Q**, configure for high-frequency burst detection.  
The fast probe (2s) is essential here — a 15s DNS block needs sub-5s sampling.

**Recommended `config.json` settings:**
```json
"fast_probe": {
  "enabled": true,
  "interval_sec": 2,
  "dns_domains": ["google.com", "its.ac.id"],
  "dns_timeout_sec": 2
},
"scheduler": {
  "telemetry_interval_sec": 5
}
```

> **Why dns_timeout_sec=2?**  
> The default `dig` timeout is 5s. If set to 5s and the burst lasts 15s,
> each fast probe cycle takes 5s — you'd only get 3 samples per burst.
> At 2s timeout, each cycle is ~2.2s → 6–7 samples per burst. ✅

**Start the sensor before running injection:**

```bash
# On Uno Q — start monitoring, log everything
source /opt/microuxi-venv/bin/activate
cd sensor/
python controller.py --duration 20m
```

## Part 6 — Run the Fault Injection

> Run **simultaneously** in a separate terminal on the laptop.  
> The sensor must already be running on the Uno Q before starting injection.

### Step 1: Dry run (verify commands without actually blocking)

```bash
cd fault-injection/scripts/
bash inject_s2.sh --target 10.64.88.173 --dry-run
```

### Step 2: Wait ~2 minutes after sensor starts (establish baseline)

Watch the sensor output on Uno Q. Wait until at least 20–30 fast probe
samples appear with `connectivity_ok: true` before injecting.

### Step 3: Run the actual injection

```bash
# Default: 10 bursts × (15s block + 45s gap) ≈ 10 minutes
sudo bash inject_s2.sh --target 10.64.88.173

# Custom parameters:
sudo bash inject_s2.sh \
    --target 10.64.88.173 \
    --bursts 12 \
    --on 15 \
    --off 45
```

### Step 4: Monitor what you expect to see

**On the laptop terminal during a burst (BLOCK phase):**

```bash
# In another terminal — watch DNS queries hitting the iptables rule
sudo iptables -L INPUT -n -v | grep DROP
# The packet counter should increment every ~2s during a burst
```

**On the Uno Q during a burst:**

Fast probe anomaly lines should appear:
```
[FAST ⚠ # 47] 08:05:12  wifi=UP  ping=OK  dns=google.com=FAIL  its.ac.id=FAIL
[FAST ⚠ # 48] 08:05:14  wifi=UP  ping=OK  dns=google.com=FAIL  its.ac.id=FAIL
[FAST ⚠ # 49] 08:05:16  wifi=UP  ping=OK  dns=google.com=FAIL  its.ac.id=FAIL
```

Key signature of S2 vs other events:
- `ping=OK` — ICMP to 8.8.8.8 still works (uplink is fine)
- `wifi=UP` — Wi-Fi stays associated
- `dns=FAIL` for **both domains** simultaneously
- `connectivity_ok=false`
- DNS failure is fast (~55ms) not slow (~2000ms) — because the laptop drops the query locally, not via internet timeout

### Step 5: Stop injection and stop sensor

After all bursts complete (or Ctrl+C to stop early):
- The script auto-restores iptables rules
- On Uno Q: Ctrl+C the controller
- Note the session ID from both outputs

## Part 9 — Expected Data Signature

After the experiment, the fast probe JSONL will contain entries like:

```jsonl
{"probe_type":"fast","ts":"2026-04-17T...","seq":47,"wifi_up":true,
 "ping":{"target":"8.8.8.8","success":true,"rtt_ms":74.1},
 "dns":[{"domain":"google.com","success":false,"latency_ms":55.2},
        {"domain":"its.ac.id","success":false,"latency_ms":54.8}],
 "connectivity_ok":false}
```

**Distinguishing characteristics of S2 vs other fault types:**

| Feature | S2 DNS Burst | S6 Connectivity Flap | S3 Loss Burst |
|---|---|---|---|
| `ping.success` | ✅ OK | ❌ FAIL | ❌ FAIL |
| `dns.success` | ❌ FAIL (both) | ❌ FAIL | ✅ Usually OK |
| `wifi_up` | ✅ UP | ✅ UP | ✅ UP |
| `dns.latency_ms` | ~55ms (fast reject) | ~55ms (fast fail) | ~65ms (normal) |
| `connectivity_ok` | ❌ false | ❌ false | ❌ false |

> **Why does `dns.latency_ms` show ~55ms (fast) during S2, not a 2s timeout?**  
> Because the DROP rule is on the **laptop** (the resolver itself). The Uno Q's
> DNS query reaches the laptop's NIC, gets dropped by iptables instantly, and
> the resolver (dnspython / dig process) times out against a local socket.
> There's no network round-trip to a remote server — the failure is local and
> fast. The 2s `dns_timeout_sec` config is the max allowed, not the actual time.

## Part 10 — Analysis (after experiment)

```python
import pandas as pd

# Load fast probe data
df = pd.read_json("sensor/out/fast_<session_id>.jsonl", lines=True)
df["ts"] = pd.to_datetime(df["ts"])

# Extract dns failure flag
df["dns_fail"] = df["dns"].apply(
    lambda lst: any(not d["success"] for d in lst) if isinstance(lst, list) else False
)
df["ping_ok"] = df["ping"].apply(
    lambda p: p.get("success", False) if isinstance(p, dict) else False
)

# S2 signature: DNS fails but ping is OK
df_s2_candidate = df[df["dns_fail"] & df["ping_ok"]]
print(f"S2 candidate events: {len(df_s2_candidate)}")
print(df_s2_candidate[["ts", "dns_fail", "ping_ok", "connectivity_ok"]])

# Load ground truth
gt = pd.read_csv("fault-injection/logs/ground_truth_s2_<ts>.csv")
gt["utc_timestamp"] = pd.to_datetime(gt["utc_timestamp"])
bursts = gt[gt["event"] == "BURST_START"][["utc_timestamp", "burst_id"]].rename(
    columns={"utc_timestamp": "burst_start"})
burst_ends = gt[gt["event"] == "BURST_END"][["utc_timestamp", "burst_id"]].rename(
    columns={"utc_timestamp": "burst_end"})
ground_truth = bursts.merge(burst_ends, on="burst_id")

# For each detected S2 candidate, check if it falls within a burst window
# MTTD = first detection ts - burst_start_ts
```

## Troubleshooting

### DNS is not being blocked

```bash
# Confirm iptables rule is inserted correctly
sudo iptables -L INPUT -n --line-numbers | head -10

# Confirm packets are being counted (during injection)
sudo iptables -L INPUT -n -v | grep DROP

# Check that dnsmasq is actually listening on the hotspot IP
sudo ss -ulnp | grep ':53'
```

### DNS still resolves during burst (has a cache)

The fast probe's dnspython resolver is configured with `cache=None`.
If you're testing with `dig` manually, note that `nscd` or `systemd-resolved`
may cache results from before the burst. Use:
```bash
dig google.com @10.64.88.124 +time=2 +tries=1 +norecurse
```

### Uno Q got a different IP after reconnect

The DHCP lease may have changed. Re-run `find_uno_ip.sh` and update `--target`.

### Hotspot drops connection during injection

This is unlikely but possible on very weak hardware. Check:
```bash
# On laptop — is the hotspot NIC still active?
nmcli device status
```

### After injection, DNS is slow to recover

Some DNS resolvers have negative TTL caching. Wait ~5s after `BURST_END`
and retry. The dnspython resolver with `cache=None` should recover immediately.
